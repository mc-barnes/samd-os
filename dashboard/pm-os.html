"Subject: Weekly Update — The team has been making solid progress…" (buries status at the bottom, no action required)

AI Agents
9-step build process, 8-layer architecture, and when not to build.
Build when the JTBD involves…
Complex multi-step workflows
"Look up in Salesforce, check Zendesk, post in Slack" — things people do by following a checklist.
Interacting with systems via APIs
Agent as universal remote. Create ticket, assign, prioritize, schedule follow-up.
Don't build when…
Task is simple and predictable
A well-designed UI is better.
Cost of error is catastrophic
Need extreme guardrails + human-in-the-loop.
Can't build the Trust Layer
No explainability or human escape = not ready to ship.
9-Step Build Process
1
Set Purpose and Scope
Start with JTBD, not technology. Narrower scope = better performance.
2
Create Clear Input/Output Structure
Treat agents like functions: structured inputs (JSON), predictable outputs (data objects), clear error states.
3
Write System Instructions
Spend 80% of your time here.
Role definition, behavioral guidelines, output format, edge case handling.
4
Enable Reasoning and External Actions
ReAct: Reason → Act → Observe. Start with simple if/then before complex chains.
5
Orchestrate Multiple Agents (if needed)
Most use cases don't need swarms. Research → Analysis → Writing → QA.
6
Implement Memory and Context
Conversation history + user context + knowledge retrieval. Start simple.
7
Integrate Multimedia Capabilities
Voice, image, document processing — add strategically, not by default.
8
Format and Deliver Results
Design outputs for both humans and systems.
9
Build UI or API
Best agents feel invisible. They just make things happen.
8-Layer Architecture — Build order: start with layers 4, 5, 7
1 InfrastructureCloud, databases, APIs. Agents make far more API calls than traditional apps.
2 Agent InternetIdentity, state management, inter-agent comms. Mostly DIY today.
3 ProtocolMCP = "USB-C for AI." Bet on standards. Build on MCP, not proprietary solutions.
4 ★ ToolingRAG, function calling, integrations. 5 rock-solid tools beats 50 flaky ones.
5 ★ CognitionPlanning, decision-making, error handling. Invest heavily in guardrails.
6 MemoryPersonal context, preferences, history. Start simple — advanced personalization after PMF.
7 ★ ApplicationThe actual product. Focus on ONE use case and nail it.
8 Ops/GovernanceBuild from day one. Retrofitting governance is enormously expensive.

Context Engineering
6 layers, CEO framework, and the 4D Canvas for AI feature specs.
"Context engineering is the delicate art and science of filling the context window with just the right information for the next step." — Andrej Karpathy. LLMs do not infer context magically.
6 Layers of Context
1
Intent
Triple-I: Interpret (explicit → task) · Infer (behavior signals) · Identify Gaps (missing info to retrieve)
2
User
5-P: Preferences · Patterns · Proficiency · Pacing · Purpose
3
Domain
5 pillars: Entities · Attributes · Relationships · Definitions/Rules · Lineage
4
Rules
Soft Wall (advisory: tone, brand) + Hard Wall (mandatory: schemas, permissions, safety)
5
Environment
N.O.W.: Nearby Activity · Operational Conditions · Window of Time
6
Exposition
Distillation Loop: Collect → Compress → Construct → Constrain → Check
4D Context Canvas — run this before every AI feature spec
D1 — Demand
What is the model being asked to do?
Specify: Inputs · Assumptions · Required Outputs · Success Criteria. Never leave outputs as "write a good response."
D2 — Data
What context is required?
For each piece: Source · Availability · Sensitivity · Priority. Run through all 6 layers.
D3 — Discovery
How will context be obtained at runtime?
Semantic search · Graph traversal · Precomputed. Define latency vs. quality tradeoffs explicitly.
D4 — Defense
How will it fail gracefully?
Pre-checks · Post-checks · Fallback paths · Feedback loops. Design failure states before launch.
What breaks AI systems: "just put everything in the prompt" · relying only on semantic search · assuming prompts enforce rules (they don't) · not maintaining provenance · trusting the model to figure out the structure.
AI Product Strategy
8-step framework, 3 principles, and patterns from the 5-layer stack.
3 Non-Negotiable Principles
Solve real problems, not AI problems
"What user problems can we solve better using AI?" — not "How can we add AI?"
Your data is your moat, not your model
Foundation models are commodities. Unique data + virtuous loops = competitive advantage.
Design for human-AI collaboration
Build fallback paths, transparency, and feedback mechanisms. Augment, don't replace.
8-Step Framework
Step 1
Set Clear Objectives
Find: friction points, pattern-rich problems, data-dense workflows, creative bottlenecks, expertise gaps.
Step 2
Understand AI Users
Trust thresholds · Collaboration zones · Learning curves · Fallback paths
Step 3
Identify AI Superpowers
Proprietary datasets, domain knowledge, unique user signals, network effects.
Step 4
Build Capability Stack
Foundation → Enabling → Application. Build/buy/partner on strategic importance.
Step 5
Visualize AI Vision
Visiontypes: before/after, collaboration, progressive intelligence, data flywheel.
Step 6
Define Key Pillars
Quick wins (1-3mo) · Strategic bets (3-12mo) · Option plays (exploratory)
Step 7
Quantify Impact
Account for learning curves, network effects, risk-adjusted outcomes. Impact compounds.
Step 8
Ethical Guardrails
What you won't build · Accuracy thresholds · Bias testing · Transparency standards

Data beats modelWinners have unique data or leverage customer data — not better algorithms.
Vertical focusSpecific domains consistently outperform horizontal approaches.

Human essentialBest AI products augment humans in collaborative interfaces.
Evals & LLM Judges
Systematic measurement of AI output quality. The PM's most underused skill.
A/B Testing
Real world performance
Evals
Predicted performance on known test cases
3-Step Eval Lifecycle
1
Analyze — Surface Failure Modes
Bootstrap 100 diverse traces → Open coding (read + label each) → Axial coding (cluster into binary failure types) → Label all traces 0/1 per failure mode
2
Measure — Build Automated Evaluators
Code-based (fast, cheap, deterministic) for objective checks. LLM-as-Judge for subjective nuance. One evaluator per failure mode — never combined.
3
Improve — Fix What You Found
Refine prompts first → Decompose tasks → Adjust RAG → Improve tool schemas → Add guardrails → Fine-tune (last resort only).
LLM Judge Data Splits
10–20% Training
Pool for few-shot demo examples only. Never used for evaluation.
~40% Dev
Iterate and refine judge prompt here. Never include in judge prompt.
~40% Test
Touch only ONCE at the very end. Final unbiased measure.
Corrected Success Rate Formula
θ̂ = (p_obs + TNR - 1) / (TPR + TNR - 1) Example: TPR=0.92, TNR=0.88, obs_pass=0.85 Corrected = (0.85 + 0.88 - 1) / (0.92 + 0.88 - 1) = 0.73/0.80 = 91.3% Report 91.3%, not 85%. Raw pass rate is biased by judge imperfection.
Most common mistake: a dashboard packed with "hallucination_score", "helpfulness_score" — these are distractions. Build application-specific evaluators for real failure modes you found in your traces.

1. Start with one AI capability, not ten

3. Include "learning milestones," not just "launch milestones"

Taste at Speed
The new PM skill: evaluate working software fast, kill most of it, ship survivors.
When building costs near zero, the bottleneck moves from "can we build it" to "should we ship it." The 80% kill rate is the whole point.
Traditional (8–12 weeks)
Idea → PRD → Design → Eng builds → QA → Ship
AI-era (1–2 weeks)
Idea → 5 prototypes → Evaluate → Kill 4 → Spec the survivor → Ship
The 5 Lenses (use /evaluate-prototype to run these)
1
Problem-Solution Fit
Does this solve the problem we set out to solve, or did it drift? State both in one sentence. Name the gap.
2
Interaction Cost
Count clicks to core value. Compare against user's current workflow. One added click = risky. Two or more = needs justification.
3
Edge Case Exposure
Empty input · Extreme input · Conflicting state · Wrong user type · Multi-attribute queries. Document every break.
4
Technical Debt Signal
Ask engineering: "If we ship this, what breaks in 6 months?" Prototype is disposable vs. IS what we're shipping.
5
Business Model Alignment
What specific metric does this move? Quantify the problem and the expected impact in dollars. Delightful ≠ strategic.
