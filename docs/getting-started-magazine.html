Medical device software teams spend hundreds of hours preparing for audits — writing PRDs, tracing design controls, analyzing risk, assembling the DHF. Most gaps are found late, when they are expensive to fix. SaMD-OS moves the discovery forward.

This guide walks you from a blank clone to your first agent-reviewed PRD in under thirty minutes. Everything here is a draft layer: artifacts you generate feed your eQMS of record, they don't replace it.

2 minutes

What Is SaMD-OS

SaMD-OS is a draft-layer working environment for teams shipping regulated medical device software. It is not a QMS — it is the uncontrolled workspace where drafts happen before they enter your quality system of record (Greenlight Guru, MasterControl, Qualio).

Clone & Customize

Get the repo on your machine and make it yours. The only file you must edit is CLAUDE.md in the project root — everything else works out of the box.

git clone https://github.com/mc-barnes/samd-os.git
cd samd-os

Open CLAUDE.md and fill in the following fields:

  • Team roster (names, roles, Slack and GitHub handles)
  • Device classification (Class II or Class III)
  • Predicate device or De Novo designation
  • Clinical domain (e.g., neonatal monitoring, cardiac, respiratory)
  • Submission type (510(k), De Novo, or PMA)
  • Standards — already pre-filled with IEC 62304, ISO 14971, ISO 13485, IEC 62366

Explore the Structure

The repository mirrors the document categories you already know from your quality system. Each top-level folder holds its own CLAUDE.md that provides context on demand — which keeps things fast even in long sessions.

product/ PRDs, strategy, competitive, customers
regulatory/ Design controls, risk, submissions, DHF
clinical/ Intended use, usability engineering
quality/ CAPA, complaints, audit prep
engineering/ Bugs, RFCs, IEC 62304 SDLC
team/ Onboarding, retros, decisions
.claude/skills/ All skills + agent personas
examples/ Sample artifacts from skills

Context loads on demand. You work in one folder, only that folder’s knowledge activates. This is how you stay fast at scale.

Draft Your First PRD

Tell Claude: “Write a SaMD PRD for [your device].” The prd-writer-samd skill generates a regulatory-ready product requirements document with everything your reviewers expect to see.

What the skill generates

  • Intended use and indications for use
  • Clinical requirements and workflows
  • Design inputs traceable to user needs
  • Preliminary hazard list

Also included

  • EHR integration requirements
  • Cybersecurity considerations
  • Proper YAML frontmatter
  • Output in product/prds/
“

The output lands with proper frontmatter, ready for human review. It is a draft — not a submission.

Run an Agent Review

Tell Claude: “Run regulatory-reviewer on this PRD.” The regulatory reviewer checks your draft against IEC 62304, ISO 14971, ISO 13485, and IEC 62366 — then returns structured findings you can act on immediately.

  • — Numbered findings with severity: CRITICAL, MAJOR, MINOR, OBSERVATION
  • — Standards citations for each finding
  • — A verdict: ACCEPTABLE, NEEDS REVISION, or NOT SUBMITTABLE
  • — Specific remediation guidance
Example Finding · Major

“Intended use statement does not specify patient population age range — IEC 62366-1:2015 §5.1 requires identification of intended patient group.”

What’s Next

You have a reviewed PRD. Now expand outward. The draft layer is designed for iterative deepening — each skill builds on what the others produce, and every artifact can be reviewed by every agent.

Browse examples/

Sample artifacts: PRD, design controls XLSX, risk analysis XLSX, FHIR bundle. See what good output looks like before generating your own.

Run the full review panel

All 5 agents review in parallel, most-conservative-wins. One command, five specialist perspectives.

Reverse-engineer from code

Code to SOUP register, code to design inputs, code to hazard candidates. Useful for brownfield devices.

Plan your team rollout

Read docs/adoption-guide.md for phased onboarding: pilot, expand, standardize.

Cost: ~$0.05–$0.15 per agent review. $5–$15/month for a typical team.

SaMD-OS is a draft layer, not a quality system. All artifacts require human review before entering your QMS of record. Generated documents are uncontrolled until formally reviewed and approved per your organization’s procedures.
