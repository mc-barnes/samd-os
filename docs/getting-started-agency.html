01
Step 01 — 2 min

What is
SaMD-OS?

A draft-layer working environment for teams shipping regulated medical device software. It is not a QMS. It is the uncontrolled workspace where drafts happen before they enter your quality system of record — Greenlight Guru, MasterControl, Qualio, whatever you use.

17 Skills
5 Specialist Agents
0 Audit Surprises
Skills generate — PRDs, risk analyses, design controls, FHIR bundles Agents review — regulatory, QA, safety, cybersecurity, clinical Draft layer — feeds your eQMS, does not replace it
02
Step 02 — 3 min

Clone &
Customize

Fork the repo. Open CLAUDE.md and fill in your team's context. This is the single file that orients every skill and agent to your specific device.

Terminal git clone https://github.com/mc-barnes/samd-os.git
cd samd-os

Open CLAUDE.md and fill in:

  • Team roster — names, roles, handles
  • Device classification — Class II or Class III
  • Predicate device
  • Clinical domain
  • Submission type — 510(k), De Novo, or PMA

04
Step 04 — 10 min

Draft Your
First PRD

Tell Claude: “Write a SaMD PRD for [your device]”

The prd-writer-samd skill generates a regulatory-ready PRD with everything you need to start the design control chain.

Prompt > Write a SaMD PRD for a continuous glucose monitoring
  algorithm that detects hypoglycemic episodes in
  Type 1 diabetes patients aged 6-17
  • Intended use / Indications for use — regulatory-grade language
  • Clinical requirements and workflows — domain-specific
  • Design inputs — traceable to user needs
  • Risk considerations — hazard candidates identified
  • EHR integration requirements — interoperability from day one
  • Cybersecurity considerations — threat surface scoped
05
Step 05 — 8 min

Run an Agent
Review

Tell Claude: “Run regulatory-reviewer on this PRD”

The agent reads your draft, evaluates it against the applicable standards, and returns structured findings.

“Findings caught before auditors do.”
Critical Major Minor Observation

Every review returns: numbered findings with severity, standards citations, a verdict (Acceptable / Needs Revision / Not Submittable), and specific remediation guidance.

MAJOR
Intended use statement does not specify patient population age range. Usability engineering file cannot scope intended user interactions without this boundary.
IEC 62366-1:2015+A1:2020, Clause 5.1
